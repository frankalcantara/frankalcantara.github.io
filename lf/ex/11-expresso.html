1. Introdução à Tabela de Símbolos

A Tabela de Símbolos é um tipo de dados abstrato que mapeia identificadores, chamados de símbolos, para a sua informação semântica associada. Essa é a principal estrutura de dados utilizada por um compilador, ou interpretador, para rastrear as entidades declaradas em um código fonte.

Identificador vs Símbolo

Identificador

Uma cadeia de caracteres, um lexema, como x ou minhaVariavel

Símbolo

Um nome único dentro de um contexto específico. Um identificador pode representar múltiplos símbolos distintos

Propósito Fundamental

Resolver a qual entidade um determinado uso de um identificador se refere

2. A Evolução da Tabela de Símbolos

A Tabela de Símbolos, embora conceitualmente simples, passou por uma notável evolução ao longo da história da computação, refletindo diretamente o avanço na complexidade das linguagens de programação.

1957 - FORTRAN I

Estruturas lineares simples (arrays, listas)

Complexidade: O(n) - busca sequencial

1960 - ALGOL 60

Tabelas hierárquicas para estrutura de blocos

Inovação: Escopos aninhados (begin...end)

1970s - Hash Tables

Revolução em performance

Complexidade: O(1) médio - busca constante

1990s - OOP

Hierarquias, sobrecarga, métodos virtuais

Desafio: Gerenciar heranças e polimorfismo

2000s - Era Moderna

Otimizações avançadas

  • Alocação em arena
  • String interning
  • Cache-conscious design

3. Interação com as Fases do Compilador

A Tabela de Símbolos é uma estrutura dinâmica que evolui ao longo do processo de compilação, servindo como interface entre o frontend e o backend.

1

Análise Léxica

Primeira fase - encontra os identificadores

  • Inserção inicial na tabela (apenas lexema)
  • Palavras-chave pré-carregadas
  • Operadores predefinidos
2

Análise Sintática/Semântica

Uso mais intenso - enriquecimento

  • Adiciona atributos detalhados (tipo, escopo)
  • Verificações de compatibilidade
  • Detecção de redeclarações
3

Geração de Código

Backend consome informação

  • Endereços de memória
  • Offsets em registros de ativação
  • Tamanhos de tipos

Implicação de Design

O design da estrutura deve acomodar preenchimento incremental e padrões de acesso variados de cada fase do compilador.

4. Informação Armazenada: Atributos de um Identificador

Uma entrada na Tabela de Símbolos é uma coleção de atributos associados a um símbolo. Os atributos exatos dependem da linguagem.

Nome

Lexema ou representação em string do identificador (ex: x, for)

Tipo

Tipo de dados do símbolo (int, float, struct)

Classe/Tipo

Natureza do símbolo (variável, constante, procedimento, parâmetro)

Escopo

Bloco léxico ou namespace onde o símbolo é válido

Memória

Endereço de execução, offset em registro de ativação

Outros

Visibilidade, constantes, dimensões de arrays, parâmetros de funções

6. Hash Tables: O Padrão De Facto

Por Que Hash Tables?

A preferência esmagadora por hash tables é consequência direta da carga de trabalho típica:

Buscas

Muito mais frequentes - um símbolo é referenciado centenas de vezes

Inserções

Raras - um símbolo é declarado apenas uma vez

Encadeamento (Chaining)

Cada bucket aponta para uma lista ligada

  • Robusto e comum
  • Performance degrada gradualmente
  • Lista ligada em cada bucket: O(n) pior caso
Endereçamento Aberto

Busca próximo espaço livre na própria tabela

  • Sondagem Linear: i+1, i+2, ...
  • Sondagem Quadrática: i+1², i+2², ...
  • Mitiga agrupamento primário

Agrupamento Primário

Problema específico da sondagem linear:

  1. Colisões formam clusters (aglomerados contíguos)
  2. Clusters crescem cada vez mais rápido
  3. Performance degrada de O(1) para O(n)
  4. Sondagem quadrática ajuda a mitigar

7. Estruturas Baseadas em Árvores

A principal alternativa ordenada às hash tables, usando BST autobalanceadas (AVL, Red-Black).

Vantagens da Ordenação

Percurso Ordenado

Listar símbolos alfabeticamente

Sucessor/Predecessor

Encontrar símbolos adjacentes

Consultas de Intervalo

Todos símbolos entre A e Z

Complexidade de Tempo

BST Balanceadas
  • Busca: O(log n)
  • Inserção: O(log n)
  • Remoção: O(log n)
  • Altura sempre logarítmica
Desvantagens
  • Maior sobrecarga de memória
  • Má localidade de cache
  • Pointer chasing
  • Implementação complexa

9. O Desafio dos Escopos Lexicais

Desde o ALGOL 60, as linguagens modernas suportam escopos lexicais aninhados. Uma variável declarada dentro de um bloco só é visível dentro desse bloco e em seus sub-blocos.

Modelo de Pilha de Tabelas

Abordagem amplamente utilizada: pilha de hash tables que espelha a estrutura de escopos aninhados.

Entrar Escopo

Push de nova hash table vazia no topo da pilha

Inserção

Sempre na tabela do topo (escopo local)

Busca

Do topo para a base - implementa shadowing

Algoritmo: Pilha de Dicionários

// Estrutura de dados global
pilha_de_escopos = []  // Inicia com escopo global

PROCEDIMENTO enter_scope():
    nova_tabela = criar_tabela_vazia()
    pilha_de_escopos.push(nova_tabela)

PROCEDIMENTO exit_scope():
    SE tamanho(pilha_de_escopos) > 1 ENTÃO
        pilha_de_escopos.pop()

PROCEDIMENTO declare(nome, atributos):
    tabela_atual = pilha_de_escopos.top()
    SE nome JÁ EXISTE em tabela_atual ENTÃO
        erro("Símbolo já declarado")
    SENÃO
        inserir(nome, atributos) em tabela_atual

FUNÇÃO lookup(nome):
    PARA cada tabela em pilha_de_escopos (topo→base):
        SE nome EXISTE em tabela ENTÃO
            RETORNAR atributos
    RETORNAR nulo  // Símbolo não declarado

Modelos Alternativos

Tabela Única com Encadeamento
  • Uma hash table global
  • Entradas encadeadas em lista ligada
  • Exit scope é complexo (limpeza seletiva)
Árvore de Escopos
  • Estrutura explícita em árvore
  • Escopo global como raiz
  • Busca sobe do nó atual até raiz

10. Implementação Funcional

Implementação em Python
class Symbol:
    def __init__(self, name, category, type):
        self.name = name
        self.category = category
        self.type = type

class SymbolTable:
    def __init__(self):
        self.scoped_tables = [{}]  # Começa com escopo global

    def enter_scope(self):
        self.scoped_tables.append({})

    def exit_scope(self):
        if len(self.scoped_tables) > 1:
            self.scoped_tables.pop()

    def declare(self, symbol):
        current_scope = self.scoped_tables[-1]
        if symbol.name in current_scope:
            raise NameError(f"Símbolo '{symbol.name}' já declarado")
        current_scope[symbol.name] = symbol

    def lookup(self, name):
        for scope in reversed(self.scoped_tables):
            if name in scope:
                return scope[name]
        raise NameError(f"Símbolo '{name}' não declarado")
Características C++23
  • std::unordered_map para hash tables eficientes
  • std::stack para gestão de escopos
  • std::optional para retornos seguros (C++17)
  • .contains() para verificação eficiente (C++20)
  • Implementa shadowing naturalmente
