2. Atividade A3: O Desafio da Ordenação

O objetivo é levar os alunos a deduzir, formalizar e analisar algoritmos de ordenação através da experimentação prática, aplicando princípios da neurociência cognitiva para maximizar a retenção e transferência do conhecimento.

Materiais Necessários

  • Conjunto de 7-9 cartas numeradas por grupo
  • Envelope A: cartas em ordem aleatória
  • Envelope B: cartas em "melhor caso" (quase ordenado)
  • Envelope C: cartas em "pior caso" (ordem inversa)
  • Folhas para pseudocódigo e tabela de rastreio
  • Canetas coloridas para marcação

Estrutura Temporal

Etapa 1: Descoberta Estruturada (22 min)

Descoberta Esperada:

Os alunos devem descobrir naturalmente algoritmos como Bubble Sort, Selection Sort ou Insertion Sort. O professor deve estar preparado para reconhecer e discutir as vantagens e desvantagens de cada abordagem descoberta.

Bubble Sort

Estratégia de comparação de vizinhos consecutivos. Simples mas menos eficiente para listas grandes.

O(n²) - Quadrática

Selection Sort

Encontra o menor elemento e coloca na primeira posição. Sempre tem poucas trocas.

O(n²) - Quadrática

Insertion Sort

Insere cada elemento na posição correta. Muito eficiente para listas pequenas ou quase ordenadas.

O(n²) - Quadrática

3. Atividade B3: Implementação de Algoritmos de Ordenação

Na Atividade A3 os alunos descobriram os algoritmos de ordenação mais comuns. Se o resultado foi satisfatório, esta atividade se resume à implementação dos algoritmos descobertos, a partir dos fluxogramas ou pseudocódigos criados pelos alunos.

Objetivo da Implementação

Implementar os algoritmos de ordenação descobertos na atividade anterior, analisando a eficiência e complexidade de cada um através de medição de tempo de execução e contagem de operações.

  • Material Necessário: Computadores com ambiente de programação configurado
  • Linguagem: C++23 com funções de medida de tempo
  • Estrutura: Uma função para cada algoritmo de ordenação descoberto
  • Análise: Três tipos de listas (aleatória, quase ordenada, ordem inversa)

Exemplo de Código Base em C++23

// Estrutura para armazenar os dados do CSV struct DataSet { std::vector<int> lista_aleatoria; std::vector<int> lista_80_ordenada; std::vector<int> lista_decrescente; }; // Bubble Sort void bubbleSort(std::vector<int>& arr) { int n = arr.size(); for (int i = 0; i < n - 1; i++) { bool trocou = false; for (int j = 0; j < n - i - 1; j++) { if (arr[j] > arr[j + 1]) { std::swap(arr[j], arr[j + 1]); trocou = true; } } if (!trocou) break; // Otimização } } // Função para medir tempo de execução template<typename Func> double medirTempo(std::vector<int> dados, Func algoritmo) { auto inicio = std::chrono::high_resolution_clock::now(); algoritmo(dados); auto fim = std::chrono::high_resolution_clock::now(); auto duracao = std::chrono::duration_cast<std::chrono::microseconds>(fim - inicio); return duracao.count() / 1000.0; // Retorna em milissegundos }

Bubble Sort

Algoritmo mais simples, mas menos eficiente para listas grandes. Comparação de elementos adjacentes.

O(n²) - Pior caso

Selection Sort

Comportamento consistente mas não adaptativo. Sempre encontra o menor elemento.

O(n²) - Sempre

Insertion Sort

Muito eficiente para listas pequenas ou quase ordenadas. Insere elemento na posição correta.

O(n) - Melhor caso

Análise Esperada dos Resultados

Performance por Tipo de Lista:

  • Lista aleatória: Merge Sort demonstra superioridade clara
  • Lista quase ordenada: Insertion Sort pode superar algoritmos mais complexos
  • Lista em ordem inversa: Representa o pior caso para a maioria dos algoritmos
  • Escalabilidade: Diferença dramática entre algoritmos O(n²) e O(n log n)

Insights sobre Implementação:

  • Simplicidade vs Eficiência: Algoritmos simples nem sempre são os melhores
  • Adaptabilidade: Alguns algoritmos se beneficiam de dados parcialmente ordenados
  • Uso de memória: Algoritmos in-place vs que requerem memória adicional
  • Comportamento assintótico: Como o desempenho muda com o tamanho dos dados

Momento Crucial: Descoberta da Busca Binária

A busca binária não é intuitiva. O professor deve induzir esta descoberta com a sugestão: "Observando as cartas ordenadas, existe uma forma mais eficiente?" Este é um momento pedagógico fundamental.

Busca Linear

Verifica elementos sequencialmente até encontrar o alvo. Sempre funciona, independente da ordenação.

O(n) - Linear
  • Melhor caso: 1 verificação
  • Caso médio: n/2 verificações
  • Pior caso: n verificações

Busca Binária

Divide o conjunto ao meio a cada comparação. Requer dados ordenados, mas é muito mais eficiente.

O(log n) - Logarítmica
  • Melhor caso: 1 verificação
  • Caso médio: log₂(n) verificações
  • Pior caso: log₂(n) verificações
Algoritmo Algoritmo Original Algoritmo Otimizado Diferença
Lista Aleatória 7-8 verificações N/A (não se aplica) -
Lista Ordenada Crescente 7-8 verificações 3-4 verificações 50% menos
Lista Ordenada Decrescente 7-8 verificações 3-4 verificações 50% menos

5. Atividade D3: Implementação de Algoritmos de Busca

Se os alunos descobriram efetivamente os algoritmos de busca na Atividade C3, esta fase foca na implementação dos algoritmos descobertos, utilizando os fluxogramas ou pseudocódigos criados anteriormente.

Objetivo da Implementação

Implementar os algoritmos de busca linear e busca binária, analisando eficiência através de medição de tempo de execução e contagem de comparações em diferentes tipos de conjuntos de dados.

  • Função de busca: Recebe lista e valor alvo, retorna posição (-1 se não encontrado)
  • Análise comparativa: Lista aleatória, lista ordenada crescente, lista ordenada decrescente
  • Métricas: Tempo de execução e número de comparações realizadas

Estruturas de Dados e Algoritmos

// Estrutura para armazenar os dados do CSV struct DataSet { std::vector<int> lista_aleatoria; std::vector<int> lista_ordenada_cresc; std::vector<int> lista_ordenada_decresc; std::vector<int> elementos_busca; }; // Estrutura para armazenar resultados de busca struct ResultadoBusca { bool encontrado; int posicao; int comparacoes; double tempo_ms; int elemento_buscado; }; // Busca Linear ResultadoBusca buscaLinear(const std::vector<int>& arr, int alvo) { auto inicio = std::chrono::high_resolution_clock::now(); ResultadoBusca resultado = {false, -1, 0, 0.0, alvo}; for (size_t i = 0; i < arr.size(); i++) { resultado.comparacoes++; if (arr[i] == alvo) { resultado.encontrado = true; resultado.posicao = static_cast<int>(i); break; } } auto fim = std::chrono::high_resolution_clock::now(); auto duracao = std::chrono::duration_cast<std::chrono::microseconds>(fim - inicio); resultado.tempo_ms = duracao.count() / 1000.0; return resultado; } // Busca Binária ResultadoBusca buscaBinaria(const std::vector<int>& arr, int alvo) { auto inicio = std::chrono::high_resolution_clock::now(); ResultadoBusca resultado = {false, -1, 0, 0.0, alvo}; int esquerda = 0; int direita = static_cast<int>(arr.size()) - 1; while (esquerda <= direita) { int meio = esquerda + (direita - esquerda) / 2; resultado.comparacoes++; if (arr[meio] == alvo) { resultado.encontrado = true; resultado.posicao = meio; break; } else if (arr[meio] < alvo) { esquerda = meio + 1; } else { direita = meio - 1; } } auto fim = std::chrono::high_resolution_clock::now(); auto duracao = std::chrono::duration_cast<std::chrono::microseconds>(fim - inicio); resultado.tempo_ms = duracao.count() / 1000.0; return resultado; }

Análise de Complexidade Demonstrada

Para 1000 elementos:

  • Busca Linear O(n):
    • Melhor caso: 1 comparação
    • Caso médio: 500 comparações
    • Pior caso: 1000 comparações
  • Busca Binária O(log n):
    • Melhor caso: 1 comparação
    • Caso médio: 9 comparações
    • Pior caso: 10 comparações

Vantagem da Busca Binária:

100x
mais rápida no pior caso
Restrição Importante: A busca binária só funciona em listas ORDENADAS!

Pergunta 1

Por que a busca binária só funciona em listas ordenadas?

Resposta: Precisa da propriedade de ordem para decidir qual metade descartar

Pergunta 2

Em que situações você usaria busca linear mesmo tendo dados ordenados?

Resposta: Quando a lista é muito pequena ou quando se busca múltiplos elementos

Pergunta 3

Como o custo de ordenação afeta a escolha do algoritmo?

Resposta: Se poucas buscas, ordenar + busca binária pode ser mais custoso que busca linear

6. Atividade E3: Criando os Inimigos da HP

O objetivo é levar os alunos a deduzir, formalizar e analisar o uso de pilha (stack) para avaliação de expressões matemáticas através da experimentação prática com notação polonesa reversa (RPN).

Materiais Necessários

  • Cartas de números (0-9, múltiplas cópias)
  • Cartas de operadores (+, -, ×, ÷)
  • Pilha física (caixa ou área delimitada)
  • Expressões matemáticas em cartões (3 níveis)
  • Calculadora para verificação
  • Cronômetro para medir tempo

Desafio Inicial

Apresentar expressões em notação polonesa reversa SEM explicar o que são. Deixar os alunos descobrirem o padrão.

Exemplos:
15 3 5 + ×
2 3 + 4 5 + ×
8 2 ÷ 3 + 1 -

1 Introdução da Ferramenta Pilha (20 min)

Regras da "Pilha Mágica"

  1. Número: sempre colocar na pilha
  2. Operador: tirar dois números da pilha, fazer a operação, colocar resultado de volta
  3. Resultado final: único número que sobra na pilha
120
Resultado: 120
Passo Símbolo Ação Estado da Pilha
1 15 empilhar [15]
2 3 empilhar [15, 3]
3 5 empilhar [15, 3, 5]
4 + calcular 3+5=8 [15, 8]
5 × calcular 15×8=120 [120]

Vantagem 1

Elimina parênteses: A estrutura inerente da pilha resolve precedência automaticamente.

Sem ambiguidade

Vantagem 2

Avaliação linear: Processa da esquerda para direita, sem backtracking necessário.

Eficiência O(n)

Vantagem 3

Reduz carga mental: Processo sistemático para expressões complexas sem confusão.

Cognitivo

Transfer Learning: Conversão de Notações

RPN: 3 4 2 × +
Resultado: 11
