12  Programação em Prolog: Recursão, Controle de Fluxo e Predicados Essenciais

A programação em Prolog distingue-se das linguagens imperativas pela sua natureza declarativa, na qual a lógica é expressa como um conjunto de relações, conhecidas como fatos e regras. Neste paradigma, a iteração não é alcançada por meio de construções de laços como for ou while, mas sim por meio do único mecanismo de repetição inerente à linguagem: a recursão.

Em uma linguagem declarativa, o programador define o que é verdadeiro sobre um domínio, em vez de prescrever como executar uma sequência de passos. A recursão alinha-se perfeitamente com esta filosofia. Em vez de detalhar um procedimento de laço, um programador Prolog define uma relação que se sustenta para um caso fundamental e que pode ser simplificada ou reduzida para um caso mais geral. Esta abordagem permite que problemas complexos sejam decompostos em subproblemas mais simples e auto-semelhantes, uma estratégia que é a essência tanto do pensamento recursivo quanto de algoritmos como “Dividir e Conquistar”.

12.1 Anatomia de um Predicado Recursivo: Casos Base e Recursivos

Qualquer definição recursiva em Prolog é composta por pelo menos duas partes essenciais: um ou mais casos base e um ou mais casos recursivos.

  1. O Caso Base: representa o cenário mais simples e trivial para o qual a solução é conhecida ou pode ser afirmada diretamente como um fato. Esta cláusula atua como a condição de parada da recursão. Sem um caso base bem definido, um predicado recursivo entraria em um ciclo infinito, chamando a si mesmo indefinidamente até esgotar a memória da pilha de chamadas, resultando em um erro de stack overflow.
  2. O Caso Recursivo: é uma regra que define o problema em termos de uma versão menor de si mesmo. Esta cláusula decompõe o problema, realiza uma parte do trabalho e, em seguida, invoca a si mesma para resolver o restante do problema, que agora está um passo mais perto do caso base.

A ordem destas cláusulas no código-fonte é importante e, frequentemente, pode afetar a lógica de terminação e a eficiência da busca do Prolog. A prática recomendada é declarar os casos base, geralmente fatos, antes dos casos recursivos, geralmente regras, pois isso pode influenciar tanto a lógica de terminação quanto a eficiência da busca do Prolog.

12.1.1 Estudo de Caso: Processamento de Listas com member/2

A estrutura de dados mais comum em Prolog, a lista, é ela própria definida recursivamente. Uma lista é ou a lista vazia, o caso base, ou uma estrutura composta por uma cabeça (head), o primeiro elemento, e uma cauda (tail), o restante da lista, representada como o caso recursivo. Esta definição inerentemente recursiva torna os predicados recursivos a ferramenta natural e elegante para o processamento de listas.

O predicado member/2, que verifica se um elemento pertence a uma lista, é um exemplo desta sinergia.

% Caso Base: X é membro de uma lista se for a cabeça dessa lista.  
member(X, [X|_]).

% Caso Recursivo: X é membro de uma lista se for membro da sua cauda.  
member(X, [_|T]) :- member(X, T).

A forma como a estrutura da lista é utilizada na unificação espelha perfeitamente a lógica de um predicado recursivo. A unificação desconstrói a lista, permitindo que a cabeça H seja processada, enquanto a chamada recursiva opera sobre uma sub-lista progressivamente menor, T. Este mapeamento direto entre a estrutura de dados e a estrutura do algoritmo é um princípio de design que confere ao Prolog sua expressividade para computação simbólica. Para ilustrar, a curiosa leitora pode considerar a consulta ?- member(b, [a, b, c]).:

  1. Prolog tenta unificar member(b, [a, b, c]) com o caso base, member(X, [X|_]). A unificação falha porque b não é igual a a;
  2. Prolog retrocede, backtracks, e tenta a segunda cláusula, o caso recursivo member(X, [_|T]) :- member(X, T). A unificação da cabeça da regra sucede, com X unificado a b, e T unificado a [b, c];
  3. A chamada recursiva member(b, [b, c]) é então executada como um novo objetivo.
  4. Prolog novamente tenta o caso base primeiro: member(b, [b|_]). Desta vez, a unificação é bem-sucedida;
  5. Como o objetivo foi satisfeito, a consulta original tem sucesso.

12.1.2 Estudo de Caso: Transversalidade de Grafos com ancestor/2

A recursão em Prolog não se limita a estruturas de dados explícitas como listas; ela é igualmente poderosa para explorar relações transitivas dentro de uma base de conhecimento, que pode ser vista como um grafo direcionado. O predicado caminho/2 é um exemplo clássico que demonstra esta capacidade.

aresta(a, b).  
aresta(b, c).  
aresta(c, d).

% Caso Base: Existe um caminho de X para Y se houver uma aresta direta de X para Y.  
caminho(X, Y) :- aresta(X, Y).

% Caso Recursivo: Existe um caminho de X para Y se houver uma aresta de X para Z e um caminho de Z para Y.  
caminho(X, Y) :- aresta(X, Z), caminho(Z, Y).

A recursão aqui define os passos de uma busca em profundidade, em inglês depth-first search, dfs. O caso base representa a descoberta de uma solução direta, enquanto o caso recursivo define como aprofundar a busca. O mecanismo de backtracking do Prolog gerencia automaticamente a exploração de caminhos alternativos. Quando um caminho falha, o Prolog retorna ao último ponto de escolha, frequentemente dentro de uma regra recursiva, e tenta a próxima alternativa. Assim, a recursão em Prolog não é apenas uma estrutura de controle, mas a própria maneira como o programador define a lógica de um algoritmo de busca, delegando ao mecanismo de inferência do do Prolog a complexa tarefa de gerenciar o estado da busca. Uma análise da consulta ?- caminho(a, c). irá revelar este processo:

A tentativa de usar o caso base caminho(X, Y) :- aresta(X, Y). falha, pois aresta(a, c). não é um fato. O Prolog tenta o caso recursivo. O primeiro sub-objetivo, aresta(a, Z), tem sucesso e unifica Z com b. O segundo sub-objetivo torna-se uma chamada recursiva: caminho(b, c). Para este novo objetivo, Prolog novamente tenta o caso base. A consulta aresta(b, c) tem sucesso, pois é um fato na base de conhecimento. Como a chamada recursiva teve sucesso, o objetivo original caminho(a, c) também tem sucesso.

12.1.3 Recursão: O Predicado factorial/2

Além de verificar a veracidade de relações, a recursão é o principal mecanismo para realizar cálculos. O predicado factorial/2, o exemplo mais tradicional possível, ilustra como um problema pode ser decomposto recursivamente e a solução construída durante o retorno das chamadas.

% Caso Base: O fatorial de 0 é 1  
factorial(0, 1).

% Caso Recursivo: O fatorial de N é F.  
factorial(N, F) :-  
    N > 0,  
    N1 is N - 1,  
    factorial(N1, F1),  
    F is N * F1.

Ao executar ?- factorial(3, X)., o mecanismo de inferência do Prolog constrói uma pilha de chamadas:

  1. factorial(3, F) chama factorial(2, F1).
  2. factorial(2, F1) chama factorial(1, F12).
  3. factorial(1, F12) chama factorial(0, F13).

Neste ponto, a chamada factorial(0, F13) unifica com o caso base factorial(0, 1), instanciando F13 como 1. A recursão então começa a desenrolar, e as computações são realizadas em cada nível de retorno:

  1. Em factorial(1, F12), F12 é calculado como 1 * 1, resultando em 1.
  2. Em factorial(2, F1), F1 é calculado como 2 * 1, resultando em 2.
  3. Finalmente, em factorial(3, F), F (que é a variável X da consulta original) é calculado como 3 * 2, resultando em 6.

Este processo de duas fases, decomposição durante as chamadas recursivas e construção da solução durante o retorno, é um padrão fundamental para a computação recursiva em Prolog.

12.2 O Fluxo de Execução do Prolog

Embora o mecanismo padrão de busca e backtracking do Prolog seja poderoso, há momentos em que um controle mais explícito sobre o fluxo de execução é necessário. Predicados de controle como o cut (!) e a negação por falha (\+) oferecem essa capacidade, permitindo que o programador otimize o desempenho e modele lógicas mais complexas, embora ao custo de alguma pureza declarativa.

12.2.1 O cut (!): Comprometendo-se com uma Escolha

O cut, representado pelo símbolo !, é um predicado especial que sempre tem sucesso, mas cujo propósito principal é procedural: podar a árvore de busca do Prolog, descartando pontos de escolha.

Quando o mecanismo de inferência do Prolog encontra e executa um !, ele se compromete com todas as escolhas feitas desde que a cláusula contendo o cut foi selecionada. Isso tem dois efeitos principais:

  1. Quaisquer cláusulas alternativas para o predicado atual são descartadas e não serão tentadas em caso de backtracking.
  2. Quaisquer pontos de escolha criados pelos objetivos à esquerda do cut dentro da mesma cláusula são congelados; o Prolog não tentará encontrar soluções alternativas para eles.

Essencialmente, o cut diz ao mecanismo de inferência do Prolog: Você encontrou o caminho certo até este ponto. Não procure por outras maneiras de satisfazer os objetivos que me precederam e não tente nenhuma outra regra para este predicado. O cut é uma concessão necessária ao pragmatismo; ele sacrifica a pureza declarativa para fornecer ao programador um controle manual sobre o processo de provas de teoremas, permitindo a otimização de desempenho ao podar ramos da árvore de busca que são conhecidos por serem redundantes ou irrelevantes.

12.2.1.1 Green Cuts vs. Red Cuts: Eficiência vs. Lógica

O uso do cut pode ser categorizado em dois tipos, com base em seu impacto na lógica do programa:

  1. Green Cuts: são usados puramente para otimização. Eles removem partes da busca que são logicamente redundantes, mas sua remoção não alterará o conjunto de soluções encontradas. Eles tornam o programa mais eficiente sem mudar seu significado declarativo.
  2. Red Cuts: alteram o significado lógico do programa. Sua remoção levaria a soluções diferentes, muitas vezes incorretas. Os red cuts são usados para implementar construções de controle, como condicionais, mas forçam uma leitura operacional do código, o que pode torná-lo mais difícil de entender e mais propenso a erros.

O predicado max/3 é um excelente exemplo para ilustrar um red cut usado para implementar uma lógica se-então-senão:

% Se X =< Y, então o máximo é Y. Não considere a outra cláusula.  
max(X, Y, Z) :- X =< Y, !, Z = Y.  

% Senão (se a primeira cláusula falhou antes do cut), o máximo é X.  
max(X, Y, X).

Neste caso, o cut é essencial para a correção. Considere a consulta ?- max(2, 5, 2)., onde estamos perguntando se 2 é o máximo entre 2 e 5:

  1. A primeira cláusula max(X, Y, Z) :- X =< Y, !, Z = Y. é selecionada, com X = 2, Y = 5, e Z = 2, já instanciado pela consulta;
  2. O teste 2 =< 5 tem sucesso;
  3. O cut (!) é executado, comprometendo-se com esta cláusula e impedindo o backtracking para outras cláusulas;
  4. A tentativa de unificar Z = Y agora significa tentar unificar 2 = 5 (pois Z já está instanciado como 2 e Y como 5);
  5. Esta unificação falha, pois 2 e 5 são valores diferentes;
  6. Como o cut já foi executado, o Prolog não pode retroceder para tentar a segunda cláusula;
  7. A consulta inteira falha, o que é o comportamento correto — 2 não é o máximo entre 2 e 5.

Sem o cut, após a falha em Z = Y, o Prolog retrocederia e tentaria a segunda cláusula max(X, Y, X), que unificaria com sucesso (pois X=2 e o terceiro argumento também é 2), retornando incorretamente true.

12.2.2 Negação por Falha (\+): Raciocinando em um Mundo Fechado

O Prolog opera sob a suposição de mundo fechado Closed World Assumption: qualquer coisa que não possa ser provada como verdadeira a partir da base de conhecimento atual é considerada falsa. A negação por falha é a implementação direta desta suposição. O predicado \+ Goal (ou not Goal) não significa logicamente falso, mas sim não provável.

A negação por falha é implementada usando uma técnica conhecida como a combinação cut-fail. O predicado fail/0 é um objetivo que sempre falha, forçando o backtracking. A definição de \+ (Goal) é efetivamente:

\+(Goal) :- call(Goal), !, fail.
\+(_).

Quando \+(Goal) é chamado:

  1. Prolog tenta executar Goal;
  2. Se Goal tiver sucesso, o cut (!) é executado, fixando esta escolha. Em seguida, fail é executado, fazendo com que a chamada inteira a \+ falhe;
  3. Se Goal falhar inicialmente, o Prolog retrocede e tenta a segunda cláusula, \+(_), que sempre tem sucesso.

Portanto, \+(Goal) tem sucesso precisamente quando Goal falha. E viva a lógica do mundo fechado!

12.2.2.1 Armadilhas Procedurais: Negação Lógica vs. Operacional

É importante que a esforçada leitora entenda que a negação por falha não é a negação lógica. Seu comportamento é procedural e altamente sensível à instanciação das variáveis no momento da chamada. Considere o exemplo de um predicado que descreve as preferências de Alice por hambúrgueres, onde ela gosta de todos, exceto do abacaxi_burger:

% Versão Correta
gosta(alice, X) :- burger(X), \+ abacaxi_burger(X).

% Versão Incorreta
gosta(alice, X) :- \+ abacaxi_burger(X), burger(X).

Na versão correta, a consulta ?- gosta(alice, X). primeiro encontra um hamburger, instanciando X. Em seguida, a negação verifica se esse X específico não é um abacaxi_burger.

Na versão incorreta, o primeiro objetivo é \+ abacaxi_burger(X). Como X não está instanciado, a consulta pergunta: É impossível provar que existe algum abacaxi_burger?. Como existe um na base de conhecimento, abacaxi_burger(X) tem sucesso, o que faz com que \+ abacaxi_burger(X) falhe, e a consulta inteira falha prematuramente, sem nunca encontrar nenhum hambúrguer que Alice aprecie.

Isso leva a uma regra de ouro na programação em Prolog: sempre garanta que as variáveis em um objetivo negado estejam suficientemente instanciadas antes que a negação seja chamada.

Note

Nota: A negação por falha é uma ferramenta poderosa, mas deve ser usada com cautela. Seu comportamento pode ser contra-intuitivo, especialmente para aqueles acostumados a raciocínios puramente lógicos. Sempre considere o estado de instanciação das variáveis e o contexto da consulta ao usar \+. Mais importante, sempre que possível use apenas predicados positivos para definir relações, recorrendo à negação por falha apenas quando absolutamente necessário.

12.3 Um Guia Prático para a Biblioteca Padrão do SWI-Prolog

O SWI-Prolog é uma implementação robusta e popular da linguagem, conhecida por seu rico conjunto de bibliotecas e predicados embutidos que facilitam o desenvolvimento de aplicações do mundo real. Esta seção apresenta uma seleção curada de predicados essenciais, categorizados por funcionalidade, para fornecer uma base prática para a programação.

12.3.1 Predicados de Entrada e Saída (I/O)

No mundo real, ilógico e interativo, a interação entre o programa, usuários e arquivos é fundamental. O SWI-Prolog possui um sistema de Entrada/Saída baseado em fluxos contínuos, streams, nos quais predicados simples como write/1 operam no fluxo de saída padrão. 12.3.2 Predicados Aritméticos

A aritmética em Prolog é realizada por meio de avaliação explícita, diferenciando-se da unificação. Uma expressão como 3 + 4 é apenas um termo, +(3, 4), até ser avaliada. O predicado is/2 é usado para avaliar expressões aritméticas, e os operadores de comparação numérica avaliam ambos os lados. A tabela ?tbl-predicados-aritmeticos resume os principais predicados e operadores aritméticos.

Operador/Predicado Descrição Exemplo de Uso
is/2 Avalia a expressão aritmética à direita e unifica o resultado com a variável à esquerda. ?- X is 3 + 4. → X = 7.
+, -, *, /, //, mod Funções aritméticas padrão usadas dentro de uma expressão a ser avaliada por is/2 ou um operador de comparação. ?- X is (10 * 2) // 3. → X = 6.
=:=/2 Avalia ambas as expressões e tem sucesso se os resultados forem numericamente iguais. Não realiza unificação. ?- 7 =:= 3 + 4. → true.
=\=/2 Tem sucesso se as expressões avaliadas não forem numericamente iguais. ?- 7 =\= 8. → true.
<, >, =<, >= Operadores de comparação numérica padrão. Eles avaliam ambos os lados. ?- 5 > 2 + 2. → true.
=/2 Unificação. Não avalia expressões aritméticas. ?- X = 3 + 4. → X = 3 + 4.

Note

Nota sobre a Ordem Padrão dos Termos: No Prolog, existe uma ordem canônica bem definida para todos os termos, utilizada pelos predicados de ordenação e comparação. Esta ordem segue a hierarquia:

  1. Variáveis (não instanciadas) - ordenadas alfabeticamente;
  2. Números - ordenados numericamente (inteiros e floats);
  3. Átomos - ordenados lexicograficamente;
  4. Termos compostos - ordenados primeiro pela aridade (número de argumentos), depois pelo nome do functor, e finalmente pelos argumentos (recursivamente da esquerda para direita).

Por exemplo, considerando a lista [f(a), 3, z, 1.5, f(b,c), a, f(b)]:

  • após ordenação: [1.5, 3, a, z, f(a), f(b), f(b,c)];
  • os números (1.5, 3) vêm primeiro, ordenados numericamente;
  • os átomos (a, z) vêm em seguida, ordenados alfabeticamente;
  • os termos compostos aparecem por último: f(a) e f(b) (aridade 1) antes de f(b,c) (aridade 2).

Esta ordem é fundamental para entender o comportamento de predicados como sort/2, setof/3 e operadores de comparação como @< e @>.
